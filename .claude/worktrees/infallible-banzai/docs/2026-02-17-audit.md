# Code Audit (2026-02-17)

## Scope

Audited all production code paths in the server and CLI:
- `server/write.go` — All write operations
- `server/mcp.go` — MCP tool handlers
- `server/main.go` — HTTP handlers, routing, middleware
- `cmd/things-cli/main.go` — CLI tool

## Summary

30 issues found: 5 high, 8 medium (data integrity), 2 medium (security), 15 low.

---

## HIGH — Potential Data Corruption / App Crash

### 1. `createArea` missing `xx` extension field
- **File:** `server/write.go:760`
- **What:** The server's `createArea` sends `{ix, tt, tg}` but does not include the `xx` extension field. The CLI's `cmdCreateArea` sends `{tt, ix, tg, xx}` with `xx: defaultExtension()`.
- **Risk:** Things may expect `xx` on all entity types. Missing fields are the same class of bug that caused the original crash (`sp: 0` instead of `sp: null`).

### 2. `createTag` sends `sh: ""` instead of `sh: null`
- **File:** `server/write.go:782`
- **What:** When no shorthand is provided, the server sends `"sh": ""` (empty string). The CLI correctly uses `*string` via `TagCreatePayload.Sh`, sending `"sh": null` when unset.
- **Risk:** Things may treat empty string and null differently for the shorthand field. Could cause display issues or sync failures.

### 3. `createTag` missing `xx` extension field
- **File:** `server/write.go:782`
- **What:** Same as issue 1 — the server does not include `xx` in tag creation payloads. The CLI does.

### 4. Race conditions on global `history`/`syncer`/`client`
- **File:** `server/main.go:14-18`
- **What:** Package-level globals accessed from concurrent HTTP handler goroutines with no mutex protection. Two concurrent writes both call `history.Sync()` and `history.Write()`, mutating `LatestServerIndex`. The retry-on-409 logic handles a single conflict but not concurrent corruption of the index.
- **Risk:** Write failures, corrupted server index, unpredictable behavior under concurrent load.
- **Fix:** Add `sync.Mutex` around all history operations.

### 5. Divergent wire formats between server and CLI
- **Files:** `server/write.go` vs `cmd/things-cli/main.go`
- **What:** The server uses `map[string]any` payloads with fewer fields. The CLI uses struct-based payloads (`TagCreatePayload`, `ChecklistItemCreatePayload`) with all fields including `xx`, nullable `*string` for optional fields, etc.
- **Risk:** If Things validates wire format strictly, the server's incomplete payloads could crash the app while the CLI's would not. This is the root cause pattern behind the original crash investigation.

## MEDIUM — Silent Data Integrity

### 6. `editTask` silently ignores invalid deadline format
- **File:** `server/write.go:635`
- **What:** If `time.Parse("2006-01-02", req.Deadline)` fails, the error is swallowed and no deadline is set. No error returned to the caller.
- **Risk:** User believes they set a deadline ("deadline: next-friday") but nothing happened. Missed deadlines in real life.
- **Fix:** Return an error when parse fails.

### 7. `createTask` same silent deadline parse failure
- **File:** `server/write.go:515`
- **What:** Identical to issue 6 but in the create path.

### 8. `createTask` silently defaults invalid `when` to inbox
- **File:** `server/write.go:508`
- **What:** When `parseWhen` returns `handled=false` for an unrecognized value (e.g. `"tomorrow"`), the code silently defaults to `st=0` (inbox). No error returned.
- **Risk:** User says `when: "tomorrow"` expecting a scheduled task, gets an inbox task instead.
- **Fix:** Return an error when `req.When` is non-empty and `parseWhen` returns `handled=false`.

### 9. Setting `project` in `editTask` strips Today/Upcoming schedule
- **File:** `server/write.go:640-647`
- **What:** When moving a task to a project without explicitly setting `when`, the schedule is forcibly set to anytime (`st=1, sr=nil, tir=nil`). This strips any existing Today or Upcoming date.
- **Risk:** "Move this task to Project X" silently removes the task from Today view.

### 10. `when: "none"` clears dates but not schedule state
- **File:** `server/write.go:624-626`
- **What:** `when: "none"` sets `sr=nil` and `tir=nil` but does not change `st`. A Someday task (`st=2`) remains in Someday with null dates.
- **Risk:** Ambiguous semantics. May not match user intent.

### 11. All read handlers ignore `syncer.Sync()` errors
- **Files:** `server/main.go:52,67,82,97,112` and `server/mcp.go:143,152,161,170,179,188,201,214,223,237,250,267,285,498`
- **What:** Every read handler calls `syncer.Sync()` and discards both return values. If sync fails (network error, auth expired), stale data is returned with no indication.
- **Risk:** User gets incorrect/outdated task lists without knowing.
- **Fix:** Log sync errors. Consider returning a warning.

### 12. Post-write `syncer.Sync()` errors also ignored
- **Files:** `server/write.go:566,580,593,670,685,698,711,724,737,750,769,793,823,883,913,931,948,966`
- **What:** After every write, `syncer.Sync()` is called to refresh local state. Error is ignored.
- **Risk:** User creates a task, immediately queries, task doesn't appear because refresh sync failed.

### 13. CLI `parseDate` silently ignores unparseable dates
- **File:** `cmd/things-cli/main.go:163-169`
- **What:** Returns `nil` on parse failure with no error. `--deadline 2024-1-15` (missing zero) silently sets no deadline.

## MEDIUM — Security

### 14. MCP endpoint has no authentication
- **File:** `server/main.go:353`
- **What:** The `/mcp` endpoint is completely unauthenticated. Anyone who discovers the URL can create, complete, trash tasks, or run the smoke test.
- **Risk:** Spam tasks, data deletion, unauthorized access to personal task data.
- **Fix:** Add optional bearer token auth, or implement OAuth.

### 15. `nuke-account` doesn't reinitialize global state
- **File:** `server/main.go:296-324`
- **What:** After deleting and recreating the account, the global `client`, `history`, and `syncer` still reference the old (deleted) account. The `newClient` returned by `SignUp` is discarded (`_ = newClient`).
- **Risk:** Server is completely broken after nuke until manual restart. All subsequent operations fail.
- **Fix:** Reinitialize globals with new client, or document that restart is required.

## LOW — Code Quality / Robustness

### 16. `createProject` doesn't use `parseWhen`
- **File:** `server/write.go:837-847`
- **What:** Hard-coded switch for `when` values. Doesn't support YYYY-MM-DD dates or validate bad input, unlike `createTask`.

### 17. Type inconsistency in `schedule()` calls
- **File:** `server/write.go:679`
- **What:** `schedule()` accepts `any` for `sr`/`tir`. Sometimes called with `int64`, sometimes `*int64`, sometimes `nil`. Works but fragile.

### 18. `generateUUID` could produce short UUIDs
- **File:** `server/write.go:182-197`
- **What:** Leading zero bytes in UUID produce fewer Base58 characters. The `isBase58UUID` validator requires `len >= 20`. Extremely rare but the generator could produce a UUID that fails its own validator.

### 19. MCP `createArea` tags not individually trimmed
- **File:** `server/mcp.go:436-440`
- **What:** Tags split from comma-separated string but not trimmed before passing to `createArea`. Downstream `validateUUIDSlice` handles it, but inconsistent with `createTask` flow.

### 20. No UUID validation on MCP read handlers
- **File:** `server/mcp.go:196-294`
- **What:** Write handlers validate UUIDs, read handlers don't. Bad UUIDs on reads just return empty results but inconsistent.

### 21. `searchTasks` has no result limit
- **File:** `server/mcp.go:493-514`
- **What:** Loads all tasks, linear scan, no cap. Searching for a common word could return a massive response causing timeouts or excessive token usage.

### 22. Read endpoints accept any HTTP method
- **File:** `server/main.go:40-124`
- **What:** `handleSync`, `handleInbox`, `handleToday`, etc. accept POST, DELETE, PUT. Should be GET-only (or POST for sync).

### 23. `jsonResponse` ignores `Encode` error
- **File:** `server/main.go:20-23`
- **What:** `json.NewEncoder(w).Encode(data)` error is discarded. Unlikely in practice but could send partial JSON.

### 24. CLI writes don't pre-sync before writing
- **File:** `cmd/things-cli/main.go:772`
- **What:** Relies on single sync at startup. Long-running batch operations risk 409 conflicts without retry.

### 25. CLI checklist items written one-at-a-time
- **File:** `cmd/things-cli/main.go:746-770`
- **What:** Each item is a separate HTTP request. Partial failure leaves task with incomplete checklist. Should use variadic `history.Write(items...)`.

### 26. CLI `--scheduled` flag has no past-date validation
- **File:** `cmd/things-cli/main.go:363-372`
- **What:** Server validates past dates for `when` and `deadline`, but CLI's `--scheduled` flag does not.

### 27. CLI `cmdShow` prefix matching too permissive
- **File:** `cmd/things-cli/main.go:661-669`
- **What:** Matches tasks by UUID prefix with no minimum length. Single character could match wrong task.

### 28. CLI batch edit can't clear fields
- **File:** `cmd/things-cli/main.go:1328`
- **What:** No "none" sentinel value support for clearing notes, deadlines, etc. in batch mode. Server's `editTask` supports this.

### 29. `noteChecksum` uint32-to-int64 cast
- **File:** `server/write.go:98-99`
- **What:** `crc32.ChecksumIEEE` returns `uint32`, cast to `int64`. Probably fine but worth noting if Things uses signed 32-bit internally.

### 30. (Duplicate of 11) MCP read handlers specifically
- **File:** `server/mcp.go:143+`
- **What:** Listed separately to call out both HTTP and MCP handler paths.
